## セキュリティ委員会に説明するときの要点

- **完全オフライン動作**：初回読み込み後はインターネット接続なしで全機能が動作します。
- **外部送信ゼロ**：入力データを外部サーバへ送る機能は実装していません。アクセス解析・エラー収集（Google Analytics / Sentry 等）も一切組み込んでいません。
  - 技術的根拠：HTML の CSP（Content-Security-Policy）で **通信先を自分自身（`connect-src 'self'`）のみ** に制限しています。
- **データは端末内のみ**：アプリ本体のデータは端末内の **IndexedDB** に保存。クラウド同期・外部 DB は使いません。
- **個人情報の入力を想定しない設計**：アプリは起動時（月1回程度）に「フルネーム・患者 ID・生年月日・住所・連絡先・画像は入力しない」旨の注意を表示します。
- **オープンソース**：ソースコードは [GitHub](https://github.com/snishi-code/snishi-code.com) で公開、第三者が検証可能です。

## QR コードのセキュリティ

このアプリの QR には2種類あり、**目的によって暗号化の有無が異なります**。

| QR の種類 | 暗号化 | 用途 |
| --- | --- | --- |
| 患者画面の QR（[05](05_電子カルテに転記する.md)） | **平文** | 電子カルテ端末の標準カメラ（Windows 11 など）で読み取り、表示された平文を電子カルテへ貼り付けるため、平文である必要がある |
| ホーム／プロブレムリスト／共有／設定／フォーマット QR（[07](07_端末間で共有する.md) / [11](11_カスタマイズ（設定）.md)） | **暗号化** | 端末間データ交換用 |

- 暗号化は **AES-GCM 256bit（改ざん検知付き）＋ 圧縮**。**汎用の QR スキャナで偶発的に医療情報が読み取られること** を防ぐのが目的です。
- ただし鍵はアプリに内蔵されているため、**厳密な秘匿（解析耐性）を保証するものではありません**。脅威モデルは「QR スキャナでの偶発的読み取り」です。
- 端末間 QR は **原則として同じ版どうし** で読み取れます（送受信でアプリの版を揃えてください）。
- **取り扱い注意**：暗号化されていても、QR の画像やテキストを保存・転送する場合は、医療情報の取り扱いに準じて慎重に扱ってください。

> 患者画面の転記用 QR は平文です。電子カルテ端末の標準カメラで読み取り、表示された内容を電子カルテへ貼り付ける運用を想定しているため、暗号化できません。**個人を特定できる情報を入力しない**運用と併せてご利用ください。
>
> 現時点では、電子カルテ端末に追加ソフトを入れず、Windows 11 標準カメラで QR を読み取る運用を想定しています。カメラ利用の可否は施設のシステム担当者にご確認ください（2026-06-10 の確認後に確定します）。

## QR の再配布制限

他端末から QR で受け取ったデータを、**さらに別の端末へ送り直せるか** は QR の種類ごとに既定値が決まっています。

| QR の種類（kind） | 再配布の既定 | 意味 |
| --- | --- | --- |
| ホーム QR（HM・患者名簿） | restricted | 受信した患者（外部由来）は再送信時に除外 |
| プロブレムリスト QR（MM） | restricted | 同上 |
| 共有 QR（SH） | free | 制限なし |
| 設定 QR（ST） | free | 制限なし |
| フォーマット QR（FMT） | free | 制限なし |

- 仕組み：他端末から受信した患者データには内部的に「外部由来（external）」マークが付きます。再配布が `restricted` の種別（ホーム／プロブレムリスト）では、QR 生成時に外部由来の患者を除外します。自端末で作成したデータ（マークなし）は制限を受けません。
- これにより「又聞きの名簿・プロブレムリストが際限なく再配布される」事態を防ぎます。申し送り（共有）は引き継いで回す想定のため `free` です。
- 暗号化・再配布の既定はコード側で定義され、常時オン（ユーザー向け設定 UI には露出していません）。ユーザー向けの説明は [07_端末間で共有する](07_端末間で共有する.md) を参照。

## アプリの更新（重要：インストール後は中身が変わらない設計）

このアプリは PWA ですが、**一度インストールしたら、配信元（サーバー）から後で中身を書き換えられない** 設計です。これは利便性より **完全性（integrity）** を優先したセキュリティ上の判断です：万一コードに不具合があっても、あるいは配信環境やアカウントが乗っ取られて悪性コードが配信されても、**すでにインストール済みの端末には一切影響しません**。

- Service Worker は登録しますが、**即時更新（skipWaiting / clients.claim）も、自動更新チェック（registration.update）も行いません**。インストール済みアプリはオンラインでも自動更新されません。
- そのトレードオフとして **正規の修正も自動では届きません**。新しい版にするには「**ホーム画面からアプリを削除 → もう一度追加**」します（＝利用者が明示的に再インストールして、初めて新しい版を信頼する形）。
- ⚠️ **削除（アンインストール）すると、その端末内のデータはすべて消えます**。更新の前に必ず JSON 保存でバックアップしてください（[09](09_データの保存と復元.md)）。

### 版を固定する・以前の版に戻す

- [GitHub](https://github.com/snishi-code/snishi-code.com) の **タグ**（`hospital-rounds-v<バージョン>`）から、任意の版に戻せます。
- 院内で版を固定したい場合は、特定タグの成果物を配布・ホスティングしてください（自前ホスティングは後述）。

## 複数端末の設定・名簿をそろえる（運用）

1台を「基準」にして、他端末へ設定・名簿を配ることができます（管理者専用モードのような特別機能はありません。通常機能の組み合わせです）。

- **設定（フォーマット・タグ・クリア対象）をそろえる** → 設定 QR（[11](11_カスタマイズ（設定）.md)）
- **フォーマット1つを配る** → フォーマット QR（[11](11_カスタマイズ（設定）.md)）
- **患者名簿を配る** → ホーム QR（受け取り側は新しい病棟として作成、[07](07_端末間で共有する.md)）
- **まとめて配る・退避する** → JSON バックアップ（現ユーザーの全病棟＋設定、[09](09_データの保存と復元.md)）
- **端末を丸ごと退避・移行する** → 「端末まるごと書出／取込」（全ユーザー、[09](09_データの保存と復元.md)）

> 旧版にあった「管理機能（管理端末/名簿配布の専用モード）」は撤去されています。上記の通常機能で代替してください。

## 多言語化（i18n）

ユーザーに見える文字列は **言語ファイル1つ** に集約されています。多言語化は翻訳ファイルを追加するだけです。

- 日本語：`src/strings.ja.json`（`"キー": "文言"` の対応表。`{name}` 形式のプレースホルダあり）
- 追加方法：`src/strings.<言語>.json` を作って翻訳 → `src/i18n.js` のロケール判定で読み込みを切り替え → 再ビルド。
- UI 側は文字列をハードコードせず、`data-i18n` 属性や `t("キー")` で展開しているため、**コード本体を触らずに言語を増やせます**。

> 説明書（この HTML 群）の多言語化は、説明書 HTML を翻訳版で用意すれば対応できます。

## 自前ホスティング・フォーク

- 本アプリは **単一の静的サイト**（PWA / 単一 HTML）です。外部 API に依存しないため、**任意の静的ホスティングに置けます**。
- フォークして自施設のサーバに設置することも可能です（ライセンス条件に従ってください）。
- **本番／テストの判定はホスト名の規約ベース** です：`dev.` で始まる・`*.pages.dev`・`localhost` 等は「テスト扱い」、それ以外は「本番扱い」。ドメインを直書きしていないため、設置先が変わっても判定は維持されます。
- 本番とテストで PWA のマニフェスト・アイコンを切り替える仕組みが入っています（テスト版アプリを本番と別アイコンで併存させられる）。

## ユーザー機能・利用ログ・巻き戻しの扱い（プライバシー設計）

1台を複数人で使い分ける **ユーザー機能**、研究用の **利用ログ**、誤操作に備える **巻き戻し（スナップショット）** を実装しています。いずれも **端末内・外部送信ゼロ** は維持されます。

- **ユーザー機能はアクセス制限ではありません**：パスワードは持たず（後から追加できる器だけ確保）、目的は「人ごとのデータ切り替え」です。誰でも切り替えられる前提で運用してください。データは人ごとに名前空間で分離（病棟・設定が別）。
- **利用ログは無記名**：1イベント＝`{時刻, ユーザーID（端末内の乱数）, 種別}` のみ。**患者名・患者IDは記録しません**。種別は起動・前面化/背面化・ユーザー/病棟切替・編集・記録クリア・QR表示・巻き戻し等。研究用の利用状況把握が目的で、「誰の記録を開いたか」は残しません。**365日で自動間引き**。
- **巻き戻し（スナップショット）は患者データを含むため短期**：破壊操作（クリア／移動／取込）と画面遷移の直前に控えを取り、**14日で自動失効**。利用ログ（無記名）とは別保管にしています。
- **3者は独立したストアに分離**：それぞれ別の IndexedDB に閉じ、相互に混ざりません（監査・消去がしやすい）。利用ログと巻き戻しは設定画面から個別に書き出し／消去できます。

> いずれも CSP（`connect-src 'self'`）下で動き、外部へ送る経路はありません。「全データを消去する」で3者まとめて消去されます。

## データの所在（技術メモ）

- **IndexedDB（3つの独立 DB）**：
  - `hospital-rounds` … アプリ本体（ユーザー登録簿・病棟・患者・SOAP・ユーザー別設定）。
  - `hospital-rounds-eventlog` … 研究用の無記名利用ログ（患者名なし）。
  - `hospital-rounds-snapshots` … 巻き戻し用スナップショット（患者データを含む・14日失効）。
- **localStorage**：初回起動マーカー・現在のユーザー／病棟を指すポインタなど **数バイトの小さな値のみ**。
- いずれも端末内に閉じています。バックアップ（JSON 書き出し）以外に外部へ出る経路はありません。
