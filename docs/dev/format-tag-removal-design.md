# フォーマットのタグ「解除」設計メモ（実装保留）

状態: **設計のみ / 実装は保留**。現状フォーマットは「適用時にタグを**付与**する」だけ（`format.tags[]` を患者タグへ merge。外す処理は無し）。本メモは将来「適用時にタグを**外す**」を入れる場合の設計と、今は入れない理由を残す。

## なぜ今は入れないか（fail-closed の観点）

「CT予定」フォーマットでタグを付与し、「CT読影」フォーマットで同じタグを外す——という運用は便利に見えるが、**外す側が「いま外してよいか」を判定できない**。

具体的には、あるタグが
- 手動で付いたのか、
- 別フォーマットの付与で付いたのか、
- まだ有効な予定を表しているのか

を区別せずに一律で削除すると、**意図しないタグ消失**が起きる。特に血液検査のように**同じフォーマットを繰り返し適用**するものでは、前回の付与・今回の付与・手動付与が混在し、誤って外す事故が起きやすい。タグ消失は臨床情報の欠落（= サイレントなデータ欠け）であり、[CLAUDE.md「データ操作は fail-closed」](../../CLAUDE.md) の精神に反する。よって**安易な自動解除は入れない**。

## 将来実装するときの方針

1. **付与と解除を明示的に分離する**: データモデルで `addTags[]`（現 `format.tags`）と `removeTags[]` を**別フィールド**にする。1 つの配列に符号付きで混ぜない（読み取り側で解釈が割れる）。
2. **UI も「追加するタグ」「外すタグ」に分ける**: 編集画面のタグピッカーを 2 枠にする。どちらに入れたかが一目で分かる形にし、片方への入れ間違いを避ける。
3. **自動推論しない＝フォーマットごとの明示アクション**: 「予定が有効か」をアプリが推測しない。外すのはユーザーがそのフォーマットに `removeTags` として明示登録したタグだけ。付与経緯（手動/別フォーマット）は問わず、**設定された removeTags に限り**外す、という単純で予測可能な規則にする。
4. **適用時の挙動**: 付与（add）と解除（remove）が同一フォーマットに同居する場合の順序・競合を定義する（例: remove → add の順で適用し、同名が両方にあれば add 優先、等）。`features/formats.js` の `applyFormatTags()` を add/remove 両対応へ拡張する想定。
5. **互換性・周辺影響を別セッションで確認**: 着手前に最低限、
   - スキーマ（`format` の新フィールド）と `normalize*()`（最新スキーマ前提で `removeTags` を補完）、
   - QR wire format（`qr-protocol.js` の「QR Wire Format Authority」コメント + 各 `WIRE_V`。フォーマット QR / 設定全体 QR に新フィールドを載せるか、短キー・enum 拡張の互換ルール）、
   - settings export / 取込（fail-closed 取込経路への影響）

   を一通り洗い出してから実装する。QR・共有形式に触れるため、本体実装とは**別セッション**で扱う。
